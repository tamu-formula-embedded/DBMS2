# Bridge

**Source:** `Core/Src/dbms/bridge.c`, `bridge.h`

## Overview

The Bridge module handles the low-level UART communication layer between the STM32 and the TI BQ79600 bridge chip. The BQ79600 translates UART frames from the STM32 into the TI COM bus protocol used by the BQ79616 monitor chips. This module is responsible for frame construction, CRC appending, baud rate control, and wake/shutdown signaling.

The Bridge module is the physical transport layer. It knows nothing about register semantics — those are handled by the [Stack](stack.md) module, which calls Bridge functions to send/receive raw frames.

---

## Hardware Context

```
STM32 UART TX/RX
      │
      └─── BQ79600 (Bridge Chip)
                │
           TI COM Bus (differential, transformer/capacitor isolated)
                │
     ┌──────────┼──────────┐
  BQ79616    BQ79616    BQ79616  ...
  (Monitor)  (Monitor)  (Monitor)
```

The BQ79600 acts as a bus master. It converts UART frames (sent by the STM32) into COM bus transactions and relays responses back over UART. It also passes through fault signals via the `nFAULT` line.

---

## Frame Format

All frames sent over UART to the BQ79600 follow this structure:

```
[ CMD | DEV_ADDR | REG_ADDR | DATA... | CRC_HIGH | CRC_LOW ]
```

- **CMD**: Command byte encoding the frame type (single read/write, broadcast read/write, etc.)
- **DEV_ADDR**: Target device address on the COM bus (0 = bridge, 1..N = monitor chips)
- **REG_ADDR**: 16-bit register address on the target device
- **DATA**: 1 or more data bytes depending on register width and command
- **CRC16**: Modbus CRC16 computed over the entire frame payload, appended as the last 2 bytes

The Bridge module appends the CRC and clears it after transmission, so the Stack module can treat frames as plain data arrays.

---

## Frame Types

The command byte encodes the frame type. Key types used:

| Frame Type | Direction | Description |
|-----------|-----------|-------------|
| Single device read | STM → BQ79600 → specific monitor | Read register(s) from one device |
| Single device write | STM → BQ79600 → specific monitor | Write register(s) to one device |
| Broadcast write | STM → BQ79600 → all monitors | Write same register to all devices simultaneously |
| Broadcast read | STM → BQ79600 → all monitors | Read register from all devices (responses daisy-chained back) |

---

## Wake and Shutdown Signaling

The BQ79616 monitor chips can be put into low-power sleep and woken up via special signals on the COM bus. Because the BQ79600 bridges UART to COM bus, these signals are generated by briefly switching the UART to an unusual baud rate that creates a continuous low or high state on the line:

| Signal | Baud Rate Used | Purpose |
|--------|---------------|---------|
| **Wake blip** | 25,700 baud | Creates a prolonged low-level signal that wakes sleeping monitors |
| **Shutdown blip** | 128,000 baud | Creates a high-speed transition that commands monitors to sleep |

The normal operating baud rate is restored immediately after the blip. These baud-rate tricks are necessary because the COM bus protocol uses break/idle conditions for wake/sleep, and the BQ79600 passes them through from UART.

---

## Timeouts

| Parameter | Value |
|-----------|-------|
| `STACK_SEND_TIMEOUT` | 50 ms |
| `STACK_RECV_TIMEOUT` | 50 ms |

If a response is not received within `STACK_RECV_TIMEOUT`, the frame exchange is considered failed and an error is returned.

---

## CRC Handling

CRC16 (Modbus polynomial 0xA001) is computed over the full frame payload before transmission and appended as the final two bytes. On receive, the CRC is verified over the received payload. Frames failing CRC are counted in `stats.n_rx_stack_bad_crcs` and discarded.

The CRC bytes in the frame buffer are zeroed out after use so the buffer can be reused safely without CRC contamination.

## Interactions

| Module | Interaction |
|--------|-------------|
